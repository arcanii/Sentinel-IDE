# PRD Quality Review — SentinelIDE (scoped: Code Signing update)

*Scope: this review judges the just-applied code-signing UPDATE (§2.3 UJ-5; §4.8 FR-19..21; §5 secrets pillar / SEC-5 / SEC-4 split; and the consistency edits in §3, §6, §12, §14, §15, §16, §17). Whole-PRD issues are flagged in passing but the verdicts weight the change and its integration. The load-bearing constraint under test is HONESTY: v1 genuinely signs the artifact, but the signing-KEY surface is not yet hardened, and the PRD must never overclaim that Sentinel protects the key.*

## Overall verdict

The signing update is unusually disciplined: the honesty line — *"the artifact is genuinely signed; the key-handling surface is not yet hardened"* — is stated identically and load-bearingly across the UJ, the FR group, the threat model, an NFR (SEC-5), a counter-metric (SM-C3), the risk register, an open question, and the assumptions index, so a reader cannot reach the feature from any entry point without meeting the caveat. FR-19..21 each carry testable consequences including the critical negative case (FR-20: an artifact that fails to sign "stays unsigned" and is never reported as signed), and the cert-store/HSM omission is explicit in three places. The one real soft spot is done-ness leakage at the edges: FR-21's "distinct trust color" is asserted but never defined in the PRD's own §9 palette (it lives only in the UX DESIGN file), and the signing feature ships with no Success Metric of its own — SM-C3 guards against overclaiming but nothing positively measures that signing works or is used.

## Decision-readiness — strong

A decision-maker can act on this. The central trade-off — ship signing UX now vs. wait until the key surface is hardened — is stated *as a decision*, not buried: §4.8 description ("import / sign / status UX ships in v1; the signing-**key** (secret) handling surface is **hardened in the next increment**"), §14 risk ("Severity: accepted for v1, honestly stated"), and OQ-5 which keeps the genuinely-open part open (the *timeline* of hardening and *which* further secrets follow) without rhetorically answering it. The `[NOTE FOR PM]` at §5 sits exactly where the tension is — buyer messaging — and gives the concrete script ("say *'the artifact is signed; the key surface hardens next,'* never *'keys are protected by Sentinel'*"), which is what a PM actually needs at a green-light gate. The choice of what was *given up* is named (key hardening, cert-store/HSM sources, an extension model), not smoothed to neutral. This is the opposite of the rubric's red flag.

## Substance over theater — strong

The update earns its content. UJ-5 drives real decisions: it forces the status-bar indicator (FR-21), the Sign-on-Build/Sign-now split (FR-20), and the in-dialog honesty note (FR-19) — none is furniture. The threat model's Pillar 2 was previously abstract ("holds secrets"); the update makes it concrete and falsifiable ("**v1 makes this concrete:** ... the IDE now actively holds a signing key — a *named secret surface* that is **not yet hardened**"), which is substance, not theater. SEC-5 is a product-specific honesty boundary with named verification (enumerate the key-handling surface for security review each release), not boilerplate. The SEC-4 vs FR-19..21 disambiguation is doing genuine work: it prevents the reader from conflating the IDE's *own* release provenance (roadmap) with the *developer's* artifact signing (v1) — a conflation a lazier PRD would have left to cause exactly the overclaim this product can't afford.

### Findings
- **low** "Distinct trust color" is asserted, not defined in-PRD (§4.8 FR-21) — FR-21 says `"Signed" uses the distinct trust color — **not** coral`. The PRD's own §9 palette names only window `#161618` and coral `#D97757`; the trust color (`#7FB37A`, `trust-verified`) exists only in the UX DESIGN file. The §17 index correctly tags it a derived UX `[ASSUMPTION]`, so this is not a missing-disclosure problem — it is a thin in-PRD definition that makes FR-21's color contract un-self-contained. *Fix:* add the trust-verified swatch (with its `[ASSUMPTION]` tag) to §9 so the §4.8 color contract resolves inside the PRD.

## Strategic coherence — strong

The update strengthens the thesis rather than diluting it. The PRD's bet is "the app is the argument" (§1) — security-critical software, including the dangerous parts, built in Sentinel — and signing is positioned precisely as the moment Pillar 3 (supply-chain apex) "becomes something Devon *does*, not just something the IDE claims" (UJ-5 climax). Crucially, the feature does **not** quietly inflate the proof: SM-C3 and the §5 note explicitly fence the signing UX *out* of the proof claim ("The 'Signed' signal must never read as 'the key is Sentinel-protected'"), so a flashy new feature is prevented from contaminating the narrow, true differentiator. That restraint is the coherence win — the update adds user-facing value while holding the line that the *only* hardened surface in v1 is still the untrusted-input parser (FR-13). Feature priority follows the thesis: the signing *experience* ships, the signing-key *hardening* is deferred because hardening order follows the threat model, not ease.

## Done-ness clarity — adequate

FR-19..21 are mostly unforgiving in the right way. FR-19 has a concrete accept set (`.pfx`/`.p12`/PEM + passphrase), a positive consequence (displays subject/issuer), a disclosure consequence (visible honesty note), and an explicit backlog fence. FR-20 carries the single most important testable consequence in the whole update — the negative path: *"On signing failure ... the artifact **stays unsigned** and the cause + fix are surfaced — the IDE never reports an artifact as signed when it is not."* That is exactly the done-ness an engineer needs and it directly enforces the honesty constraint at runtime. FR-21 enumerates all six indicator states discretely (`no key · key loaded · signing · signed · unsigned · failed`) with the "exactly one live state" invariant — testable.

Two gaps keep this from `strong`:

### Findings
- **medium** Signing has no positive Success Metric (§15) — the update added counter-metric SM-C3 (don't overclaim) but no primary/secondary SM that signing works or is adopted. SM-1 ("prefers SentinelIDE ... for the edit→build→run loop", Validates FR-1..FR-11) explicitly stops at FR-11 and does not cover FR-19..21; SM-2 is the proof metric, which signing is deliberately fenced out of. So the feature ships graded only by what it must *not* claim, with nothing measuring that it delivers. *Fix:* add a secondary SM (e.g., "a loaded key produces an Authenticode-signed executable verifiable via Windows file properties; signing-failure surfaces cause+fix") validating FR-19..21 — the verification handle already exists in the UX memlog (`file properties > Digital Signatures`).
- **low** FR-21 color/severity contract is partly an adjective without an in-PRD bound — `"unsigned" is neutral (not alarmist)`, `"failed" uses the error color`. "Error color" and "neutral" resolve only via the UX DESIGN palette, not §9. Same root as the FR-21 color finding above; grouped here because it touches done-ness of the indicator's visual contract. *Fix:* covered by adding the relevant swatches to §9.

## Scope honesty — strong

This is the update's strongest dimension and the one the constraint most depends on. Omissions are explicit, not inferred:
- The key-hardening deferral is stated as a `[NON-GOAL for MVP]`-equivalent in §12.2 ("**Hardening** of the signing-key handling surface — next increment ... but key handling is not yet a Hardened surface, SEC-5") and again in the §11/§12 framing.
- The cert-store / HSM / smart-card omission is fenced in **three** places: FR-19 consequence ("**Backlog (not v1)**"), §12.2 ("Signing-key sources beyond key files ... are **backlog**"), and §17 index. No reader can silently assume these.
- Every signing inference is tagged and indexed: §17 carries FR-19 key-source, SEC-5 "v1 UX harden next" split, and the derived `trust-verified` color, each pointing back to its inline use. The Assumptions-Index roundtrip holds for the new entries (checked below).

Open-items density is appropriate for a green-light-to-build PRD: the signing update added exactly one OQ (well, updated OQ-5), one NOTE-FOR-PM line, and three indexed assumptions — proportionate to a feature that ships UX now and defers hardening. Nothing is de-scoped silently.

## Downstream usability — strong

For a chain-top PRD feeding UX → architecture → stories, the signing update sources cleanly. New glossary terms **Code signing** and **Signing key** (§3) are defined once and used identically downstream — "Authenticode," "key file (`.pfx`/`.p12`/PEM)," "native host," "not-yet-hardened," and the FR-19..21 / SEC-5 anchors recur verbatim in §4.8, §5, §12, §14, §15, §17 with no drift. Every signing cross-reference I traced resolves, including the *external* ones: EXPERIENCE Flow 5, DESIGN `status-signing` and `import-key-dialog`, and the `trust-verified` color all exist in the sibling UX artifacts (`../ux-designs/ux-SentinelIDE-2026-06-27/`), so the FR-19/FR-21 "*(DESIGN ...)*" tags are live, not dangling. UJ-5 has a named protagonist (Devon, consistent with UJ-1..3) carrying full context inline. Each signing section reads standalone via Glossary terms rather than "see above." ID continuity is intact (see Mechanical notes).

## Shape fit — strong

The shape matches the product. SentinelIDE is a developer tool with a security-buyer proof audience — a case where named-protagonist UJs are load-bearing, and the update correctly adds one (UJ-5 / Devon) rather than leaving the signing feature as a bare capability. The feature is *not* over-formalized: it is one UJ, one feature group, three FRs, one NFR, one counter-metric — proportionate. It is also not under-formalized: because the security claim is the whole product, the update correctly spends extra rigor on the §5 threat-model integration and SEC-5 rather than treating signing as ordinary chrome. The honesty disclosure being pushed *into the product UI* (FR-19 dialog note, FR-21 indicator semantics), not just the PRD, is the right shape for a product whose differentiator is "provably can't be turned against you."
