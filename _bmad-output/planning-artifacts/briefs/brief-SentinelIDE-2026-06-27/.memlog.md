- (decision) Build posture RESOLVED = B-path + C-rule + A-destination. (Pure A was considered then reframed.) B-path: ship a usable IDE on a thin NATIVE host now. C-rule: every security-relevant surface (untrusted input, secrets, build/run, output) is Sentinel from day one, no exceptions; native code allowed ONLY for security-irrelevant chrome. A-destination: native footprint is a tracked, shrinking debt driven to zero (North Star fed to language team). Rationale: the security proof lives in the SURFACES, not the chrome, so B delivers 100% of the security proof immediately; pure A would gate the security story behind months of security-irrelevant GUI plumbing.
- (decision) Purpose #4 = MIGRATION CASE STUDY. The native->Sentinel migration is a second deliverable: a documented playbook for hardening an EXISTING native (C/C++) codebase by porting exposed surfaces to Sentinel surface-by-surface — the realistic path for banks/gov with legacy code they can't rewrite. The IDE's own shrinking native footprint is the reference implementation. Only B enables this (pure A is greenfield). Reinforces the B decision.
- (decision) Purpose #3 (re-affirmed) = FORCING FUNCTION for the Sentinel language: building a real GUI app in Sentinel surfaces capability gaps (native GUI/FFI/etc.) fed back as prioritized requirements to the language team; demand-pull (B) produces sharper requirements than spec-push (A). Full 'why' is a QUADRUPLE: proof + tool + forcing-function + migration case study.
- (decision) Differentiation = apex supply-chain hardening. An IDE sits on Sentinel's exact 3 threat classes AND at the TOP of a supply chain: (1) executes untrusted code constantly, (2) holds secrets, (3) is the top of a supply chain (everything authored through it inherits its integrity). Honest day-one claim under B: 'every surface touching your data/secrets/build-output is Sentinel; remaining native code is shrinking window chrome — here's the number' — stronger to a CISO than an unverifiable '100% Sentinel'.
